## Risks

R1
- Risk: 回填腳本之正則表達式或 idempotency 判斷有缺陷，致 24 個既有 artifact 之 Metadata 區段重複插入 `PDCA Stage` 欄位、或誤改其他欄位順序，污染整個 registry
- Trigger: 對任一 artifact 之 Metadata 區段使用非 anchored regex（如 `Status:` 比對到任一行而非 metadata 內），或重跑 backfill 時未檢查既有 `PDCA Stage` 行
- Detection: `git diff` 中 plan artifact 出現重複 `PDCA Stage:` 行；或 `guard_status_validator.py --task-id TASK-XXX` 對既有任務報出 metadata 結構錯誤
- Mitigation: 回填腳本必須 (a) 以 `^- PDCA Stage:` anchored 檢查 idempotency；(b) 先在 `--dry-run` 模式對全 24 任務輸出 diff 預覽；(c) 提交前以 `git stash` 備份；發生錯誤即 `git checkout -- artifacts/` 還原並修腳本
- Severity: blocking

R2
- Risk: root 與 template/ 之兩層架構文件 / TAO schema / validator 任一處同步缺漏（如新增 `docs/agentic_execution_layer.md` 而忘記新增 `template/docs/agentic_execution_layer.md`），致 [guard_contract_validator.py](artifacts/scripts/guard_contract_validator.py) 在下次 CI 跑時失敗，全 repo 處於 contract drift 狀態
- Trigger: 在 Phase 1-3 修改檔案後，跳過 Phase 4 直接進 Phase 5；或 Phase 4 漏列任一檔
- Detection: `python artifacts/scripts/guard_contract_validator.py` 報 `[FAIL]` 含 `template/` 之路徑；或 `diff` root 與 template 對應檔差異
- Mitigation: Phase 4 完成後立即跑 contract validator 全盤掃描；發現 drift 即同步補檔；Phase 5 不啟動直至 Phase 4 全綠
- Severity: blocking

R3
- Risk: TAO Trace 之欄位設計（`Thought Log` / `Action Step` / `Observation` / `Next-Step Decision`）對實際 subagent dispatch 不夠具體，致 Codex / 後續 subagent 在填寫時各自詮釋、產出格式不一，未來 validator 無法 lint
- Trigger: [docs/agentic_execution_layer.md](docs/agentic_execution_layer.md) 僅以散文描述 schema，未附最小範例與 BNF 風格欄位定義
- Detection: 自驗證階段抽核 1-2 份新 dispatch 之 code artifact，TAO Trace 欄位有 ≥ 2 種不同格式
- Mitigation: schema 文件須含至少 2 個完整範例（一個 risk ≥ 3、一個 risk = 2）；每欄位附 `expected length`、`required keywords`；Phase 3 validator 加上 `--check-tao` 之 strict-mode 之雛形（即使預設 warning-only，亦規定欄位名稱必須完整）
- Severity: non-blocking

R4
- Risk: [classify_existing_tasks.py](artifacts/scripts/classify_existing_tasks.py) 對既有 plan artifact 之 R1-Rn severity 推定錯誤（如 plan artifact 文字未嚴格遵守標準 severity 行格式），致 risk classification 結果與實際不符，high-risk 任務未補 TAO Trace
- Trigger: 既有 plan artifact 中 Severity 欄位文字含全形冒號、額外空格、或被引用於說明文字而非 risk 條目
- Detection: 對 [risk_classification.csv](artifacts/registry/risk_classification.csv) 抽核 5 條，與 plan artifact 之 Severity 欄位人工比對
- Mitigation: classifier 同時印出每條 risk 之原文片段供人工抽核；對推定為 high-risk 之任務，補 TAO Trace 前再人工確認；對推定錯誤者，記入 known-issues 並手動覆寫
- Severity: non-blocking

R5
- Risk: Phase 5 大量改動 24 個既有 task 之 plan/code/verify artifact 後，其中部份 task 之 status.json 之 `last_updated` 與 artifact 之 `Last Updated` 不一致，致 [guard_status_validator.py](artifacts/scripts/guard_status_validator.py) 之 cross-check 規則報出大量 warning，淹沒 signal
- Trigger: backfill 腳本只更新 artifact metadata 之 `Last Updated`，未同步更新 status.json
- Detection: 跑 `python artifacts/scripts/guard_status_validator.py` 對所有 24 任務，看到 `[WARN] last_updated mismatch` 數量遽增
- Mitigation: backfill 腳本同步更新對應 status.json 之 `last_updated`；或加旗標 `--skip-status-update` 並在文件中註明此次回填為「歷史回顧」型，不視為 status 變動
- Severity: non-blocking

R6
- Risk: PowerShell wrapper [Invoke-GeminiAgent.ps1](artifacts/scripts/Invoke-GeminiAgent.ps1) 之 retry 邏輯在 Gemini CLI 寫檔成功後仍回非零退碼時，會 9 次重試（3 model × 3 attempt），重複寫入同一 artifact 並消耗外部 API quota；本日已實際觀察到此現象（見 task notification by9n1xw0y）
- Trigger: 任一 dispatch 之 gemini.cmd 在成功寫檔後 `$LASTEXITCODE != 0`（行為待研）
- Detection: 背景任務 notification 顯示 failed 但 artifact 已生成；或 dispatch log 顯示 9 次相同 attempt
- Mitigation: 後續 dispatch 採 `run_in_background=false` + 較短 timeout，並於 wrapper 加 early-exit（artifact 已成則不重試）；本任務僅一次 research dispatch 已完成，後續 Phase 不另派 Gemini，影響可控
- Severity: non-blocking
