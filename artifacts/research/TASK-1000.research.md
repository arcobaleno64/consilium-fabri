## Research Questions
- PDCA 在軟體專案管理（Software Project Management）之既有實作方式？
- ReAct / TAO 在 multi-agent 系統（如 LangGraph, AutoGen）之 prior art 與設計模式？
- 宏觀（Macro, 專案級）與微觀（Micro, 代理人級）循環共存之設計範例與架構？

## Confirmed Facts
- **PDCA 於 AI 程式碼生成之應用**：InfoQ 提出了一個專門針對 AI 程式碼生成的 PDCA 框架，將 Plan 定義為需求分析與 Prompt 設計，Do 為代碼生成，Check 為測試與驗證，Act 為根據回饋修正 Prompt 或代碼。這與 `consilium-fabri` 目前的 workflow（Intake -> Research -> Planning -> Coding -> Verification -> Closure）高度吻合 [https://www.infoq.com/articles/PDCA-AI-code-generation/]。
- **ReAct (Reasoning and Acting) 核心機制**：ReAct 透過在模型生成的過程中交替產生「推論軌跡」（Reasoning Traces）與「特定任務動作」（Task-specific Actions），顯著提升了 LLM 在複雜決策中的表現與可解釋性。這種模式允許模型在行動前進行思考，並根據環境觀察（Observation）更新其內部的推論過程 [https://arxiv.org/abs/2210.03629]。
- **TAO (Thought-Action-Observation) 於 Agentic Loops 之角色**：TAO 是實現 ReAct 的一種具體實作模式，廣泛應用於現代 Agent 框架。Thought 代表內部的推論，Action 是對外部工具或系統的操作，Observation 是操作後的反饋。這種循環構成了代理人的「執行層」 [https://www.emergentmind.com/topics/reason-act-reflect-react-architecture]。
- **LangGraph 的分層循環管理**：LangGraph 透過「狀態圖」（State Graphs）支援巢狀循環（Nested Loops）。宏觀循環可以是一個高階的狀態機（如 PDCA），而微觀循環（如 ReAct 代理人）則作為圖中的一個節點或子圖存在。這種結構允許明確定義層次間的通訊與控制轉移 [https://docs.langchain.com/oss/python/langgraph/workflows-agents]。
- **AutoGen 的多代理人協作模式**：AutoGen 透過多個代理人之間的對話來解決複雜任務。在層次化設計中，可以由一個「管理者代理人」（Manager Agent）負責 PDCA 的決策與監控，而「執行代理人」（Executor Agent）則在其內部運行 TAO 循環來執行具體指令 [https://www.microsoft.com/en-us/research/blog/autogen-enabling-next-generation-large-generation-llm-applications/]。
- **層次化協作預測控制 (Hierarchical Coordinated Predictive Control)**：在工業製程中，多代理人系統常採用層次化架構，上層負責長週期的規劃與目標設定（類似 PDCA 的 macro-control），下層負責短週期的即時執行與回饋（類似 TAO 的 micro-execution） [https://www.mdpi.com/2076-3417/14/14/6025]。
